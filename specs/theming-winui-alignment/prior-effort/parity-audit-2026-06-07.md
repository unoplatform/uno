# Theming WinUI Parity Audit — Synthesis Report

## 1. Executive Verdict

The Uno theming subsystem is **at or very near 1:1 behavioral parity with WinUI** across all nine audited subsystems for the in-scope Skia targets. Of the **17 candidate gaps** surfaced by the per-subsystem comparisons, adversarial verification **refuted 12** (verdict=`not-real`), **confirmed 4** (verdict=`real`), and left **1 uncertain**.

What this means concretely:

- **The hot paths are clean.** ThemeResource resolution, owner-theme threading, the Enter-establishment walk, the NotifyThemeChanged walk, app/OS theme precedence, StaticResource-alias resolution, owner-less fallback, and popup/flyout inheritance all converge with WinUI on resolved value, ActualTheme, event ordering, and propagation. Most refuted claims rested on a misread of *WinUI* (e.g., assuming WinUI bakes a global active theme, or fires ActualThemeChanged synchronously before `m_theme` is assigned — it is actually posted async), not a real Uno defect.

- **The remaining confirmed gaps are all LOW severity and cluster on one axis: High Contrast.** Three of the four confirmed gaps are HC-related, and the common root cause is that Uno collapses the HC dimension to a single boolean sourced from a settable feature flag, with **no OS high-contrast detection wired on Skia at all**. There is no high/medium-severity correctness defect on the Light/Dark base-theme path.

- **No confirmed gap blocks parity for default/Light/Dark scenarios.** Every confirmed gap requires either (a) an OS or manual HC transition, or (b) a deliberately conflicting non-stock authored resource, to observe.

Classes of gaps remaining: (i) High-Contrast variant fidelity & event surface; (ii) one structural keyframe-scope gate that is value-neutral for stock templates but theoretically divergent with a hostile ancestor override.

## 2. CONFIRMED Gaps (verdict=real), by corrected severity

All four confirmed gaps carry **corrected severity = LOW**. There are no high/medium confirmed gaps. Ordered by breadth of observable impact.

### 2.1 — App-wide HC variant selection derives from base theme, not the OS HC variant (White/Black collapse)
- **Subsystem:** highcontrast-composition · id `hc-variant-collapsed-to-bool-appwide-path`
- **WinUI ref:** `dxaml/xcp/components/theminginterop/SystemThemingInterop.cpp:121-177`; `dxaml/xcp/core/core/elements/Resources.cpp:738-753`
- **Uno ref:** `src/Uno.UI/UI/Xaml/ThemeResolution.cs:68-69`; `src/Uno.UI/UI/Xaml/ResourceDictionary.cs:668-686`
- **Observable divergence:** With distinct `HighContrastWhite` and `HighContrastBlack` theme dictionaries authored, app base theme = Dark, no element `RequestedTheme`, and the OS in a White-on-black HC variant: WinUI selects `HighContrastWhite` (keyed off the literal OS variant via `GetSystemHighContrastTheme`); Uno computes `GetHighContrastKeyForBaseTheme(Dark) = HighContrastBlack` and resolves the **wrong** sub-dictionary. Symmetric for OS-Black + base Light. Reachable today only via the manual `WinRTFeatureConfiguration.Accessibility.HighContrast` flag plus distinct dictionaries — cannot yet arise from genuine OS HC state (no Skia OS-HC detection).
- **Suggested fix:** Carry the real OS HC variant end-to-end. (1) Extend `ISystemThemeHelperExtension.skia.cs` with a method returning the OS HC variant (None/White/Black/Custom), reading the platform HC color scheme (analog of `GetSystemHighContrastTheme`). (2) Surface the variant on `SystemThemeHelper` and thread its bits into the effective Theme. (3) In `ResourceDictionary.ResolveActiveThemeDictionary`, when no subtree theme is requested, select the HC sub-dictionary from the OS variant (mirroring `Resources.cpp:740-753`); only fall back to `GetHighContrastKeyForBaseTheme` on the subtree-RequestedTheme branch. Add a runtime test (distinct White/Black dicts, base Dark, OS variant forced White → assert White brush).

### 2.2 — `HighContrastCustom` sub-dictionary is unreachable in Uno
- **Subsystem:** highcontrast-composition · id `hc-custom-unreachable`
- **WinUI ref:** `dxaml/xcp/core/core/elements/Resources.cpp:750-752`; `dxaml/xcp/components/theminginterop/SystemThemingInterop.cpp:170-173`
- **Uno ref:** `src/Uno.UI/UI/Xaml/Theming.cs:24`; `src/Uno.UI/UI/Xaml/ResourceDictionary.cs:685-686,1132-1156`
- **Observable divergence:** `Theme.HighContrastCustom` exists only as an enum constant (`Theming.cs:24`) and is never produced or consumed; there is no `Themes.HighContrastCustom` key and `ResolveActiveThemeDictionary` never emits it. An app authoring a `HighContrastCustom` theme dictionary (plus a generic `HighContrast` fallback) running under a custom (non-white/non-black) OS HC scheme: WinUI selects `HighContrastCustom`; Uno can never select it and falls back to base-derived White/Black or the generic `HighContrast` key. **Scope note (correction from verification):** on WinUI's *subtree-requested-theme* path WinUI itself maps Light→White / Dark→Black and never selects Custom, matching Uno — so the divergence is confined to WinUI's app-wide path. Already tracked as accepted follow-up work (parity-performance-review.md:83-84,156; code comment at `ResourceDictionary.cs:663-667`).
- **Suggested fix:** Same end-to-end OS-variant plumbing as 2.1: classify White/Black/Custom in system-theme detection, expose an enum-valued accessor in place of the bool, add a `Themes.HighContrastCustom` key (`ResourceDictionary.cs:1141-1143`), and in `ResolveActiveThemeDictionary` select `HighContrastCustom` before falling back to the generic `HighContrast` key. Leave as documented deferred work until OS-HC detection exists on Skia; do not block on it.

### 2.3 — High-contrast transition does not re-resolve already-applied `{ThemeResource}` values (no OnThemeChanged-equivalent walk)
- **Subsystem:** app-os-precedence-customtheme · id `explicit-app-theme-hc-toggle-no-rewalk`
- **WinUI ref:** `dxaml/xcp/components/theming/FrameworkTheming.cpp:43-62`; `dxaml/xcp/core/dll/xcpcore.cpp:8057-8089`
- **Uno ref:** `src/Uno.UWP/FeatureConfiguration/WinRTFeatureConfiguration.Accessibility.cs:15-26`; `src/Uno.UI/UI/ViewManagement/AccessibilitySettings.cs:59-69`
- **Observable divergence:** On an HC toggle, WinUI's `OnThemeChanged` refreshes `m_highContrastTheme`, flips `GetTheme()`, and forces a full tree walk (`fForceRefresh=true`) re-resolving every applied `{ThemeResource}` against the `HighContrast*` sub-dictionary. Uno's HC toggle only raises `AccessibilitySettings.HighContrastChanged` (sole subscriber is `BackdropMaterial`, a localized brush update) — nothing invokes a tree-wide re-resolution. So **newly** resolved values pick `HighContrast*` (the leaf reads `IsHighContrast` live and invalidates its cache), but **already-applied** values keep their non-HC values until the next base-theme walk or element materialization. Reachable today only via the manual config/test flag (no OS HC listener on Skia), hence low.
- **Suggested fix:** Wire `AccessibilitySettings.HighContrastChanged` (or a `SystemThemeHelper`-level HC-changed event sourced from the config flag) to the same `forceRefresh` tree walk used for base-theme changes — i.e. invoke `NotifyThemeChanged(currentEffectiveTheme, forceRefresh:true)` (`Application.cs:591`) even though the Light/Dark base key is unchanged, then raise `UISettings.OnColorValuesChanged()`. The leaf already reads `IsHighContrast` live, so only the re-walk trigger is missing. (Real OS-HC detection on Win32 is a separate, larger prerequisite.)

### 2.4 — Walk never raises `HighContrastChanged` and never applies WinUI's HC suppression/fire rules for `ActualThemeChanged`
- **Subsystem:** notify-walk-and-actualthemechanged · id `no-highcontrastchanged-no-actualtheme-suppression-on-hc`
- **WinUI ref:** `dxaml/xcp/core/core/elements/framework.cpp:3346-3386` (`RaiseActiveThemeChangedEventIfChanging`)
- **Uno ref:** `src/Uno.UI/UI/Xaml/FrameworkElement.Theming.cs:289-293`; `src/Uno.UI/UI/Xaml/Application.cs:588`; (no `HighContrastChanged` event exists on Uno `FrameworkElement`)
- **Observable divergence:** WinUI carries HC bits into the walk theme (`base|HC`); each FE's `RaiseActiveThemeChangedEventIfChanging` raises `FrameworkElement.HighContrastChanged`, and suppresses `ActualThemeChanged` while entering/staying in HC but fires **both** when *leaving* HC with a base change. Uno's `FrameworkElement` has **no `HighContrastChanged` event at all** (grep finds it only on `AccessibilitySettings`), the walk theme carries only Light/Dark, and a pure HC toggle does not even start a walk (`SetRequestedTheme` early-returns). So on any OS HC transition that doesn't flip the base, WinUI raises `HighContrastChanged` on every FE (and conditionally `ActualThemeChanged`); Uno raises neither. Unconditional and observable to any app code subscribing to `FrameworkElement.HighContrastChanged`. Low because no OS-HC listener is wired on Skia and the event is rarely consumed.
- **Suggested fix:** Two-part, worth doing only once an OS HC change is actually observable on Skia. (1) Add `public event TypedEventHandler<FrameworkElement, object> HighContrastChanged;` to `FrameworkElement.Theming.cs` and raise it mirroring `framework.cpp:3364-3376` ordering (raise `HighContrastChanged` first; suppress `ActualThemeChanged` while entering/staying HC; fall through to the base-change `ActualThemeChanged` otherwise). (2) Carry the HC dimension into the walk and start a walk on a pure HC toggle: wire `SystemThemeHelper` to an OS HC notification, track an `IsHighContrastChanging` flag analogous to `Application.IsBaseThemeChanging`, and force `OnResourcesChanged(ThemeResource)` + root `NotifyThemeChanged` when only HC changed. (Note: 2.3 and 2.4 share the same missing OS-HC-listener prerequisite and should be implemented together.)

## 3. UNCERTAIN Gaps (need manual / native-WinUI validation)

### 3.1 — Keyframe Phase-A theme refresh gate keys on the target element's activeness, then walks the keyframe DO's own scope
- **Subsystem:** themeresource-refresh-pin · id `keyframe-phaseA-gate-uses-owner-not-self` · verdict **uncertain**, corrected severity **low**
- **WinUI ref:** `D:\Work\microsoft-ui-xaml2\src\dxaml\xcp\components\DependencyObject\Theming.cpp:315-346` (gates the ancestor walk on `this->IsActive()`; a keyframe DO is not in the visual tree so it skips Phase A and only runs `RefreshValue()` against the pinned dict)
- **Uno ref:** `D:\Work\uno-worktrees\theming\src\Uno.UI\UI\Xaml\DependencyObjectStore.Theming.cs:497-511` (Phase-A gate is `owner is UIElement { IsActiveInVisualTree: true }`) + `ObjectAnimationUsingKeyFrames.cs:339-345` (passes `owner: targetElement`); `GetResourceDictionaries()` at :499 runs on `this` = the keyframe store
- **Why uncertain:** The gate divergence is **real in code** — Uno enters Phase A for keyframes (because `owner` = the live target FE is active) where WinUI does not. The original claim asserted Uno's keyframe parent chain "resolves nothing"; verification found the **opposite** — the chain *does* reach template-root and higher live-tree FrameworkElement resource dictionaries (keyframe → ObjectAnimation → Storyboard → VisualState → VisualStateGroup → template-root FE). That makes a **constructible value divergence reachable**: if any live ancestor FE `Resources` holds the same key X with a value different from the keyframe's parse-time pinned dictionary, Uno's Phase A returns the ancestor value while WinUI returns the pinned value. For stock templates both resolve identically (pinned dict == the scope the walk reaches), so it cannot be proven equivalent *or* proven to bite in practice — hence uncertain.
- **What to validate:** On native WinUI, run a Storyboard whose `DiscreteObjectKeyFrame.Value="{ThemeResource X}"` on an active control while a live ancestor `FrameworkElement.Resources` (e.g. an enclosing `Grid.Resources`) also defines X with a *different* value; observe which value WinUI applies at `Begin` vs Uno.
- **Suggested fix (if parity is desired):** In `DependencyObjectStore.Theming.cs:497`, gate Phase A on the activeness of the DO whose scope is walked (`this`), not on `owner` — e.g. `if (ActualInstance is UIElement { IsActiveInVisualTree: true })`, keeping the `owner`-derived `ownerTheme` for theme selection. This makes a non-active keyframe skip the ancestor walk and fall straight to Phase B `RefreshValue` against its pinned dictionary, matching WinUI's `this->IsActive()` gate. Behavior-neutral for all non-keyframe elements (the normal caller passes `owner == ActualInstance`).

## 4. Refuted claims (verdict=not-real) — do not re-investigate

- **`app-override-pinned-null-not-appdict`** (themeresource-refresh-pin): App-level `{ThemeResource}` overrides ARE pinned to a live Application/merged dictionary at parse time via `ResourceResolver.ApplyResource`→`TryTopLevelRetrieval`; the `!wasSet` tree-walk branch is a backstop, not the pin mechanism. WinUI pins the app *root* (process-lifetime), so its weakref is never dead; both re-query and converge.
- **`keyframe-local-themeresource-never-pinned`** (storyboard-keyframe): `immediateResolution` is NOT forced true for VSM-materialized storyboards — `GoToState` pushes the captured template `_xamlScope`, so `TryStaticRetrieval` finds and pins the template-local dictionary at materialization. Template/page-local keyframe keys DO resolve.
- **`keyframe-theme-none-bail-no-winui-analog`** (storyboard-keyframe): WinUI's `RefreshValue` is gated by `m_theme!=None || !IsValueFromInitialTheme()` (`Theming.cpp:340-343`) and performs no Begin-time refresh; its None-gate matches Uno's bail. Neither re-resolves at Begin; both snapshot.
- **`alias-target-lazy-per-theme-vs-winui-parsetime-bake`** (staticresource-alias): WinUI does NOT bake against a single global active theme — undeferral is per-sub-dictionary and consumer-theme-selected. Uno threads the same consumer theme; resolved brushes are identical (caching difference only).
- **`keyframe-resolves-against-target-not-keyframe-scope`** (enter-theme-establishment): Uno's Phase-A `GetResourceDictionaries()` runs on the keyframe store (its own ancestor chain), never on the target's ancestors; the claimed target-ancestor shadowing is impossible.
- **`keyframe-theme-none-early-return`** (enter-theme-establishment): `ResolveOwnerTheme` never returns `None` (falls back to app theme == `Themes.Active`), so removing the bail would re-resolve to the identical value; the hypothesized None-with-differing-effective-theme window is unreachable.
- **`keyframe-skipped-for-non-fe-target`** (enter-theme-establishment): Keyframe owner-theme is set at materialization (owning FE scope) and kept correct by the recursive theme walk — both target-type-independent. The early-return only skips a redundant supplemental pass.
- **`resolveownertheme-uses-store-parent-not-logical-parent`** (enter-theme-establishment): WinUI itself uses two chains (logical-aware at Enter, visual-with-popup-follow at resolution); Uno mirrors this. `ResolveOwnerTheme` short-circuits on the owner's own established `_theme`, so the chains agree; the divergent state is one Enter precludes.
- **`actualtheme-value-stale-at-event-vs-new`** (notify-walk-and-actualthemechanged): WinUI delivers `ActualThemeChanged` **asynchronously** (posted `WM_SCRIPT_CALL_BACK`, not sync), so by the time the handler runs `m_theme` is already the NEW theme — identical to Uno. Confirmed by native `ResourceDictionaryBasicTests.cpp:4209-4234`. The Uno test comment's *attribution* is loosely worded but the asserted value is correct.
- **`explicit-app-theme-os-accent-syscolor-no-rebuild`** (app-os-precedence-customtheme): Uno sources `SystemAccentColor`/`SystemColor*` from **static XAML literals**, not the live OS, in *all* theme states; a pure OS accent change raises no event on Skia. The divergence is a theme-state-independent limitation, not the explicit-app-theme gate the claim blamed.
- **`popuproot-lateadd-apptheme-drops-hc`** (popup-flyout-inheritance): The carried HC bits are functionally inert on **both** sides — WinUI's `SetRequestedThemeForSubTree` strips HC to base immediately, and both engines re-derive HC live at the leaf. No path reads the carried HC bits to choose a dictionary.
- **`menuflyout-items-theme-timing-at-opener-enter`** (popup-flyout-inheritance): Claim inverts reality. WinUI's MenuFlyout-item Enter is a *dead* enter (`fIsLive=FALSE`), so it does NOT theme items at opener-Enter; Uno DOES (via `PropagateThemeEnterToDPPropertyValues` reaching `MenuFlyout.Items`). Uno is a superset, not a regression.
- **`doc-says-themes-active-impl-uses-actualelementtheme`** (ownerless-fallback): `ActualElementTheme` and `Themes.Active` both flow exclusively from `Application.RequestedTheme` in production, so they are always equal — a documentation-prose nit, not an observable behavior gap.
- **`two-ownerless-fallback-sources-diverge-only-under-setactivetheme`** (ownerless-fallback): The two owner-less fallback sources diverge only via the **test-only** `ResourceDictionary.SetActiveTheme` API (no production caller sets `Themes.Active` independently of `RequestedTheme`); it is a stale-test-contract artifact, not a WinUI-parity gap.
